# telegram_pair 작업 가이드

이 디렉터리에서 작업할 때는 상위 `../AGENTS.md`를 따르되, 아래 로컬 규칙을 추가로 적용한다.

## 프로젝트 개요
- 목적: Telegram 그룹 채팅에서 Claude/Codex 로컬 CLI를 오케스트레이션하는 Python 패키지
- 런타임: Python 3.10+
- 주요 패키지: `telegram_pair/`
- 테스트: `tests/`

## 작업 원칙
- 변경은 작고 되돌리기 쉽게 유지한다.
- 기존 패턴과 타입 힌트를 우선 재사용한다.
- 새 의존성은 사용자 요청 없이는 추가하지 않는다.
- 동작 변경이 있으면 관련 테스트를 함께 갱신하거나 추가한다.

## Python 모듈 크기 가이드
- `.py` 모듈은 **가능하면 500줄을 넘기지 않도록 제안**한다.
- 모듈이 500줄에 가까워지면(예: 400줄+) 먼저 책임 분리를 검토한다.
- 모듈이 **500줄을 초과하면 모듈화 방안부터 고안**한 뒤 구현한다.

## 권장 모듈화 방식
- 역할별 분리: 예) `telegram_app.py` → handlers / sender / progress / commands
- 도메인별 분리: 예) orchestration / routing / config / models / adapters
- 입출력 분리: CLI 호출, Telegram I/O, 순수 비즈니스 로직 분리
- 공통 로직 추출: 반복되는 검증/포맷팅/파싱은 util 또는 helper 모듈로 추출

## 현재 상태 메모
- 현재 500줄을 넘는 `.py` 파일은 없다.
- 비교적 큰 파일은 다음과 같다.
  - `telegram_pair/orchestrator.py` (~378줄)
  - `telegram_pair/telegram_app.py` (~355줄)
- 위 파일들은 기능 추가 시 500줄 제한을 넘지 않도록 우선 분리 후보로 검토한다.

## 분리 트리거 예시
- 한 파일에 클래스/핸들러/포맷터/외부 연동이 함께 섞일 때
- 테스트에서 한 파일의 fixture/검증 범위가 지나치게 넓어질 때
- import 없이 내부 private 함수만 계속 늘어날 때
- 섹션 헤더가 4개 이상으로 늘고 책임이 명확히 갈릴 때

## 검증
- 코드 변경 후 가능하면 `pytest`로 회귀를 확인한다.
